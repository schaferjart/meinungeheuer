# Plan 07-01: Live Concept Map Visualization

**Goal:** Replace the chat transcript in `ConversationScreen` with a dynamic, generative concept map that evolves in real-time during the AI voice conversation.

**Approach:** Client-side concept extraction from transcript entries, force-directed graph layout, hybrid Canvas (lines) + DOM (text labels) rendering, evolving definition text.

---

## Task 1: Concept Extractor (`conceptExtractor.ts`)

**What:** A pure function that takes a string (transcript text) and returns extracted concepts/keywords.

**Implementation:**
- Use lightweight client-side NLP. Options in order of preference:
  1. **compromise.js** (~200KB) — excellent noun/topic extraction, works in browser
  2. **Custom regex + stopword filter** — zero dependencies, less accurate but tiny
  3. Start with option 2 (custom) for minimal bundle size. Can upgrade to compromise later.
- Custom approach: split text into words, lowercase, remove stopwords (common articles, prepositions, pronouns), remove words < 4 characters, return unique nouns/terms.
- Maintain a curated stopword list for both German and English (the installation is bilingual).
- Return type: `ConceptExtractionResult = { concepts: string[]; rawText: string }`
- Each concept is a normalized lowercase string (for deduplication) with a display form (original casing).

**Tests:**
- Extracts meaningful nouns from English sentence
- Extracts meaningful nouns from German sentence
- Filters stopwords in both languages
- Handles empty/whitespace input gracefully
- Deduplicates concepts
- Ignores very short words

**Files:** `apps/tablet/src/lib/conceptExtractor.ts`, `apps/tablet/src/lib/conceptExtractor.test.ts`

## Task 2: Concept Map State Hook (`useConceptMap.ts`)

**What:** React hook that manages the concept graph state: nodes, edges, evolving definition, lifecycle.

**Input:** `transcript: TranscriptEntry[]` (same prop already passed to ConversationScreen)

**State shape:**
```typescript
interface ConceptNode {
  id: string;              // normalized concept text
  label: string;           // display text (original casing)
  firstSeen: number;       // timestamp
  lastSeen: number;        // timestamp (updated on re-mention)
  mentionCount: number;    // how many times mentioned
  opacity: number;         // 0-1, managed by lifecycle
  x: number;              // position (from force layout)
  y: number;
  sourceRole: 'visitor' | 'agent' | 'both';
}

interface ConceptEdge {
  source: string;          // node id
  target: string;          // node id
  weight: number;          // co-occurrence count
}

interface ConceptMapState {
  nodes: ConceptNode[];
  edges: ConceptEdge[];
  definitionDraft: string; // evolving definition text
}
```

**Logic:**
1. On each new `TranscriptEntry` added to transcript:
   - Call `extractConcepts(entry.content)` to get concepts from this turn.
   - For each concept: if node exists, update `lastSeen` and increment `mentionCount`. If new, create node with initial position near center (jittered).
   - For each pair of concepts within the same turn: create or strengthen an edge (increment weight).
   - Cap at 30 visible nodes. When adding node 31+, fade out the oldest/least-mentioned node.
2. **Lifecycle tick** (run on `requestAnimationFrame` or `setInterval(1000)`):
   - Any node whose `lastSeen` is >60 seconds ago begins fading (reduce opacity by 0.02/tick).
   - Nodes reaching opacity 0 are removed from state.
3. **Definition draft:**
   - Scan agent transcript entries for the most recent substantive statement (>50 characters, not a question).
   - Update `definitionDraft` when a better candidate appears.
   - Simple heuristic: longer agent statements that contain the term are preferred.

**Tests:**
- Adding a transcript entry creates concept nodes
- Repeated concepts increment mentionCount
- Co-occurring concepts create edges
- Edge weight increases on repeated co-occurrence
- Node cap at 30 evicts oldest/least-mentioned
- Stale nodes (>60s) begin fading
- Definition draft updates from agent statements

**Files:** `apps/tablet/src/hooks/useConceptMap.ts`, `apps/tablet/src/hooks/useConceptMap.test.ts`

## Task 3: Force-Directed Layout (`useForceLayout.ts`)

**What:** A lightweight force simulation that positions concept nodes organically on screen.

**Algorithm:**
- **Repulsion:** Every node repels every other node (Coulomb's law). Prevents overlap.
- **Attraction:** Connected nodes attract each other (spring force / Hooke's law). Pulls related concepts together.
- **Centering:** Gentle force pulling all nodes toward the center of the viewport. Prevents drift to edges.
- **Damping:** Velocity decays each frame. System settles into stable positions but never fully freezes (add small random perturbation for organic feel).
- **Bounds:** Nodes constrained to visible area (with padding for labels).

**Implementation:**
- Pure function `simulateStep(nodes, edges, bounds) => nodes` that mutates positions.
- Hook `useForceLayout(nodes, edges)` runs simulation via `requestAnimationFrame`.
- Returns positioned nodes (same array, updated x/y).
- When new nodes are added, they enter the simulation with zero velocity near a related node or center.
- Lightweight: O(n^2) for repulsion is fine with n <= 30.

**Performance:**
- Skip simulation when no nodes are visible.
- Run at animation frame rate but only update React state every ~3 frames (throttle to ~20 position updates/sec to avoid excessive re-renders).

**Tests:**
- Nodes without edges spread apart (repulsion)
- Connected nodes cluster together (attraction)
- Nodes stay within bounds
- Adding a node triggers simulation
- Empty node list is handled gracefully

**Files:** `apps/tablet/src/hooks/useForceLayout.ts`, `apps/tablet/src/hooks/useForceLayout.test.ts`

## Task 4: Canvas Connection Lines (`ConceptMapCanvas.tsx`)

**What:** A full-screen Canvas element that draws connection lines between concept nodes.

**Rendering:**
- Canvas fills the entire conversation area (behind text labels).
- On each animation frame, clear canvas and redraw all edges.
- Edge rendering: bezier curve between node positions. Stroke color: `rgba(255,255,255, weight * 0.15)`. Stroke width: `1 + weight * 0.5` (thicker = stronger connection). Maximum width 4px.
- Optional: subtle glow effect via `shadowBlur` on canvas context (test performance on iPad).
- Canvas resizes with container (observe via `ResizeObserver`).

**Props:** `edges: ConceptEdge[]`, `nodePositions: Map<string, {x: number, y: number}>`, `containerRef: RefObject<HTMLDivElement>`

**Performance:**
- Canvas is cheap to redraw at 30fps with <100 edges.
- Use `requestAnimationFrame` synced with the force layout.

**Files:** `apps/tablet/src/components/ConceptMapCanvas.tsx`

## Task 5: Concept Node Component (`ConceptNode.tsx`)

**What:** A positioned DOM element for each concept label.

**Rendering:**
- Absolutely positioned `<span>` with `transform: translate(x, y)` for GPU-accelerated positioning.
- Text styling: monospace or serif font, white, size varies with `mentionCount` (base 0.85rem, grows to 1.3rem for heavily-mentioned concepts).
- Opacity animated via CSS transition (matches node.opacity from state).
- Entry animation: `scale(0) -> scale(1)` over 300ms with slight overshoot (CSS `cubic-bezier`).
- Exit animation: handled by opacity fade (from lifecycle in useConceptMap).
- Subtle text-shadow for depth.

**Props:** `node: ConceptNode` (includes x, y, opacity, label, mentionCount)

**Files:** `apps/tablet/src/components/ConceptNode.tsx`

## Task 6: Evolving Definition Component (`EvolvingDefinition.tsx`)

**What:** Bottom-anchored text displaying the working definition draft.

**Rendering:**
- Fixed position at bottom of conversation area, above mic indicator.
- Large serif text, brighter than concept labels (`rgba(255,255,255,0.85)`).
- When `definitionDraft` changes, crossfade: old text fades out (300ms), new text fades in (300ms).
- If no definition draft yet, show nothing (don't show placeholder text).
- Max 3 lines, ellipsis overflow for very long definitions.

**Props:** `text: string`

**Files:** `apps/tablet/src/components/EvolvingDefinition.tsx`

## Task 7: Rewrite ConversationScreen (`ConversationScreen.tsx`)

**What:** Compose all pieces into the new ConversationScreen, replacing the transcript UI.

**Structure:**
```
ConversationScreen
├── Term badge (top-left) — preserved from current
├── Concept map area (flex-1, relative positioned)
│   ├── ConceptMapCanvas (absolute, fills area — draws lines)
│   ├── ConceptNodeElement * N (absolute, positioned by force layout)
│   └── EvolvingDefinition (absolute, bottom of area)
├── Mic indicator (bottom) — preserved from current
```

**Implementation:**
1. Call `useConceptMap(transcript)` to get nodes, edges, definitionDraft.
2. Pass nodes/edges to force layout for positioned coordinates.
3. Render `ConceptMapCanvas` with edges and node positions.
4. Map over nodes to render `ConceptNodeElement` for each.
5. Render `EvolvingDefinition` with definitionDraft.
6. Keep term badge and mic indicator identical to current implementation.

**Props:** Same interface as current `ConversationScreenProps` — no changes to the component's public API. The `transcript` prop is still passed in but now feeds the concept map instead of rendering chat bubbles.

**Files:** `apps/tablet/src/components/screens/ConversationScreen.tsx`

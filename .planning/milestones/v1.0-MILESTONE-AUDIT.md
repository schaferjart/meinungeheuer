## E2E Flow (1/1 Verified)

**Text → ElevenLabs TTS → Timestamps → KaraokeReader → Word Highlighting → Playback Complete**

Full flow traced through source code:
1. `stripMarkdownForTTS(text)` prepares TTS input
2. `useElevenLabsTTS` fetches audio + character-level timestamps
3. `buildWordTimestamps` converts to word-level `WordTimestamp[]`
4. `KaraokeReader` renders words with `data-kr-index` attributes
5. `useAudioSync` rAF loop binary-searches timestamps, sets `data-kr-state`
6. CSS transitions animate word highlighting
7. `ended` event → `onComplete` → state machine advances

## Tech Debt

### Phase 01 — Package Foundation
- SUMMARY frontmatter missing `requirements_completed` for all 10 Phase 1 REQ-IDs (plans predate this convention; VERIFICATIONs have full per-REQ evidence)

### Phase 04 — Validation and Publication
- **CSS property name mismatch:** `TextReader.tsx` sets `--kr-highlight` and `--kr-content-max-width` but package CSS declares `--kr-highlight-color` and `--kr-max-width`. Override mechanism silently fails. Visual output is correct because default values coincidentally match.
- **Dual audio element:** `TextReader.tsx` creates its own `HTMLAudioElement` and calls `play()`/`pause()` directly, bypassing `useKaraokeReader`'s status machine. Works due to `autoPlay` prop in kiosk environment. If autoplay is blocked, manual Continue click would play audio but word highlighting wouldn't start (`useAudioSync` only runs when status is `'playing'`).
- **Cache adapter voice_id:** `supabaseCacheAdapter.ts` hardcodes `voice_id: 'unknown'` — generic `CacheAdapter` interface doesn't carry voiceId. Functionally correct (key includes voiceId hash), data quality limitation only.
- **Pre-existing TS errors:** `useConversation.ts` ElevenLabs SDK type mismatches (documented in CLAUDE.md, predates this milestone).
